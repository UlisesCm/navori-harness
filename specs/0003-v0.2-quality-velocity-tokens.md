# Spec 0003 — navori v0.2: calidad, velocidad, tokens

> **Estado**: draft.
> **Objetivo de release**: v0.2.0.
> **Framing**: navori es tool-for-self. El MVP ya cubre tu workflow Bonum.
> Este spec NO agrega features nuevas grandes — endurece lo que existe.
> **Origen**: lecciones destiladas de auditoría a ponytail y ECC, filtradas con
> el lente "esto me ayuda a mí en Bonum, no a popularizar".

---

## 0. Resumen ejecutivo

Tres principios rectores, en este orden de prioridad:

1. **Calidad de código** — la CLI y los assets generados deben sobrevivir 6 meses sin sorpresas.
2. **Optimización de tokens** — los harnesses que navori produce cuestan tokens en cada sesión Bonum. Reducir 30% del peso del harness rendereado es leverage compound.
3. **Velocidad** — `navori init/render/sync/doctor` corre N veces al día. Cada 500ms cuenta.

Lo que **no** se hace en v0.2:

- Multi-engine adicional (sigue Claude Code-first).
- Plugin ecosystem público formalizado.
- Selective install architecture estilo ECC.
- Branding, i18n, monetización, control plane Rust.

---

## 1. Estado actual honesto

Lo que ya está bien (no tocar):

- `navori.config.json` como source of truth única — modelo correcto, mantener.
- Marcadores `<!-- navori:managed -->` con merge model — moat real, mantener.
- Render reconstruye desde config — idempotente por diseño, mantener.
- Preset system con baseline+extras — funciona, mantener.
- Stack chico (pnpm + TS + Zod + citty) — predecible, mantener.

Lo que **se puede afilar** (este spec):

- Velocidad de render no medida ni optimizada.
- Token cost de los harnesses generados no es objetivo explícito.
- Doctor diagnostica pero no remedia ni hace invariants check.
- Tests existen pero no cubren idempotencia ni regresiones de output.
- Hooks generados no llevan wrapping defensivo por default.

---

## 2. Principios

### 2.1 Calidad de código
- Cualquier helper repetido en 2+ archivos vive en `lib/util/`.
- Schemas Zod con tests de boundary (qué passes, qué fails explícitamente).
- Errores tipados — no `throw new Error("...")` libre. Clases por dominio.
- Lint estricto: `no-any`, `prefer-const`, `no-unused-vars`. Enforced en CI.
- Coverage mínimo 80% en `packages/cli/src/lib/`.

### 2.2 Optimización de tokens (harnesses generados)
- Cada SKILL.md generado tiene un **cap de palabras** declarado por tipo.
- Cada SKILL.md tiene `description` con triggers explícitos → carga on-demand, no always-on.
- Bloques `<!-- managed -->` vacíos o solo-comments se compactan a una línea.
- Rules per-language auto-pruning: si el proyecto es TypeScript-only, no se rendea `rules/python.md`.
- Hooks no inyectan contexto al session salvo que el plugin declare `injectsContext: true`.

### 2.3 Velocidad
- Render incremental con hash cache (`.navori/cache.json`, gitignored).
- I/O paralelo donde aplique (escrituras independientes, file reads en detection).
- Lazy plugin loading: solo parsear manifests de plugins que se vayan a usar.
- `NAVORI_BENCH=1` emite timings por step.
- Targets: `init` < 2s, `render` < 500ms repos chicos, < 2s para webapp.

## 3. Áreas de mejora (ordenadas por leverage)

### 3.1 Quality gates (P0)

**3.1.1 Invariants check en doctor**
Inspirado en `scripts/check-rule-copies.js` de ponytail.

Cada plugin/preset declara en su manifest:
```json
{
  "invariants": [
    "Boundaries section present",
    "No commits to .claude/",
    "Conventional commits in español MX"
  ]
}
```

`navori doctor` valida que cada string aparece literal en el output rendereado. Falla CI si alguno desaparece. **50 LOC**. Previene la clase entera de "el refactor del template se comió una regla".

**3.1.2 Idempotency test**
`navori render` dos veces seguidas → diff vacío bit-a-bit. Hoy probablemente lo cumple por construcción; en v0.2 se vuelve test explícito en `packages/cli/src/__tests__/idempotency.test.ts`. Si rompe, es regresión clase A.

**3.1.3 `navori render` por default es preview; `--apply` escribe**
Decisión Q1: cambio de comportamiento. `navori render` ahora muestra qué archivos se crearían/modificarían + diff por archivo, sin escribir. Para mutar disk se requiere `navori render --apply`. Fricción intencional: previene sorpresas al 100% sin importar el tamaño del cambio.

Migración: el behavior anterior queda disponible vía `navori render --apply` explícito. Documentar en CHANGELOG como breaking change v0.1 → v0.2.

**3.1.4 `sync --interactive` (conflict resolution)**
Cuando el sync detecta que una región managed fue editada a mano, en vez de fallar o pisar:
- Muestra el conflicto (3-way: anterior managed, actual del repo, nuevo render).
- Opciones: keep mine / accept new / merge / edit manually.

Esto cierra el caso "Ulises editó CLAUDE.md a mano y el sync siguiente perdió el cambio".

**3.1.5 `doctor --fix` (auto-remedy)**
Cuando doctor detecta drift remediable (managed block faltante, hook sin wrapping defensivo, plugin instalado sin sus assets), ofrece patch automático opt-in. Si no lo aceptas, sigue como warning.

**3.1.6 Catalog count enforcement (de ECC, modesto)**
Doctor cuenta archivos en `skills/`, `agents/`, `presets/` del repo `navori-harness` y los compara contra los counts en `package.json#features`. Falla CI si no matchean. Cuesta 20 LOC, previene drift de docs.

**3.1.7 Pre-commit hook opt-in scaffolded por navori**
Decisión Q5: no `doctor --strict` en CI, sí pre-commit hook **opcional** local.

`navori init` ofrece scaffoldear `.husky/pre-commit` (o equivalente sin husky) que corra `navori doctor --strict`. El hook detecta drift antes del commit y bloquea localmente. Si el dev hace `git commit --no-verify`, pasa — CI no lo bloquea.

Razonamiento: tu flujo en Bonum es controlado y Conventional commits atómicos. Atrapar drift antes de commit reduce ruido en PRs sin imponer fricción en CI cuando hay urgencia. Opt-in para no asumir que todos los repos Bonum quieren el mismo gate.

### 3.2 Token optimization en harnesses generados (P0)

**3.2.1 Output discipline por tipo de skill**
Cada SKILL.md tiene un `type` declarado en frontmatter: `behavior | reference | tool`. Cada tipo tiene cap de palabras:

| Tipo | Cap body | Cuándo aplica |
|---|---|---|
| behavior | ≤200 palabras | Skills que dictan cómo actúa el agente (ej. tdd-workflow) |
| reference | ≤500 palabras | Skills que documentan un patrón (ej. mantine-patterns) |
| tool | ≤300 palabras | Skills que envuelven una herramienta externa (ej. bun-runtime) |

Doctor falla si una skill excede su cap sin override explícito.

**3.2.2 Triggers explícitos en `description`**
Convención obligatoria en SKILL.md generados:
```yaml
description: >
  Use when the user says "review for over-engineering",
  "what can we delete", or invokes /review-cleanup. Reviews
  diff for unnecessary complexity.
```

Los triggers permiten que Claude Code cargue la skill **on-demand**, no always-on. Reduce contexto base de cada sesión.

**3.2.3 Variantes inline filtrables (recipe doc only en v0.2)**
Decisión Q3: documentar el patrón sin implementar el filter en v0.2.

Crear `docs/recipes/skill-variants.md` con:
- El problema (skills con modos sin querer mantener N archivos).
- El patrón (variantes inline + filtrado regex en render).
- 1 ejemplo concreto.
- Cuándo agregar el filter (si aparece preset Bonum con caso real).

Implementación (40 LOC + tests) queda para v0.3 cuando exista el caso. Hoy en v0.2 nadie en Bonum tiene preset con modos, no se justifica.

**3.2.4 Managed block compaction**
Bloques `<!-- managed -->` vacíos se renderean como una línea:
```html
<!-- navori:managed:skills:start --><!-- navori:managed:skills:end -->
```
en vez de un placeholder con N líneas. Save ~50-100 tokens por sección vacía.

**3.2.5 Rules per-language auto-pruning**
`detect.ts` ya detecta lenguajes. El render solo incluye `rules/<lang>.md` para los detectados. Hoy probablemente ya lo hace; v0.2 lo formaliza con test.

> **v0.2 — NO APLICA.** Al mapear el código no existe ningún `rules/<lang>.md`: la arquitectura es preset-based, no language-based, así que no hay rules de otros lenguajes que podar. Construir un sistema de rules sería una feature nueva grande, no "afilar lo que existe". Queda para v0.3 si aparece el caso. (El modelo preset-based ya es selectivo: un proyecto solo recibe los assets de su preset.)

**3.2.6 Hook context discipline**
Por default, hooks SessionStart **no** inyectan contexto. Solo si el plugin declara `injectsContext: true` explícitamente. Evita el escenario "el hook escupe 2k tokens de instrucciones a cada sesión silenciosamente".

> **v0.2 — CUMPLIDO POR CONSTRUCCIÓN.** El schema de hooks solo soporta `PreToolUse | PostToolUse | Stop`; no existe `SessionStart` ni ningún hook que inyecte contexto (los hooks bundleados son PreToolUse de verificación: quality-gate, jscpd, semgrep). El problema que este item previene no existe en navori. Agregar `SessionStart` + `injectsContext` sería superficie especulativa (anti-objetivo del spec). Se reactiva cuando aparezca un hook que realmente inyecte contexto.

**Métrica:** medir tokens del harness rendereado en `bonum-webapp` antes/después. Target: -30%.

### 3.3 Velocidad (P1)

> **Cierre de Fase 3 — medido primero (3.3.4 antes que todo).** Con `NAVORI_BENCH=1`
> instrumentado, la medición sobre un repo nextjs + plugins:
>
> | Comando | Medido | Target | Margen |
> |---|---|---|---|
> | render (escribe) | 67ms (plan 4ms / write 62ms) | <500ms | 7× |
> | render (sin cambios) | 6ms | — | — |
> | init --recommended | 160ms | <2s | 12× |
> | doctor | 60ms | <1s | 16× |
> | node startup (piso) | 60ms | — | — |
>
> navori **ya cumple todos los targets con holgura 7-16×**. El `plan` (lo que el
> cache 3.3.1 aceleraría) es 3-5ms; gran parte del tiempo de cualquier comando es
> arrancar node. **Decisión (Ulises, con datos): NO construir 3.3.1 ni 3.3.2** —
> 400+ LOC de complejidad por <50ms es sobreingeniería (anti-objetivo). Fase 3
> cierra con 3.3.4 (bench) + 3.3.3 (dedup plugins). Se reevalúa si una medición
> futura en un repo pesado (bonum-webapp) muestra un cuello real.

**3.3.1 Hash cache para render incremental (granular por dependencia)**
Decisión Q2: tracking granular, no todo-o-nada.

> **v0.2 — NO SE CONSTRUYE.** El `plan` mide 3-5ms; un cache de ~400 LOC
> (más provenance 3.6.2) para ahorrarlo no se justifica. La decisión Q2
> (granular vs todo-o-nada) queda moot mientras no haya cuello real. Reabrir
> solo si el bench en un repo pesado lo pide.

Archivo `.navori/cache.json` (gitignored):
```json
{
  "version": 1,
  "outputs": {
    "CLAUDE.md": {
      "hash": "sha256:abc...",
      "writtenAt": "...",
      "sources": [
        { "kind": "config", "path": "navori.config.json#core", "hash": "sha256:..." },
        { "kind": "preset", "id": "medusa", "hash": "sha256:..." },
        { "kind": "navori-version", "value": "0.2.0" }
      ]
    },
    ".claude/skills/foo/SKILL.md": {
      "hash": "sha256:def...",
      "sources": [
        { "kind": "preset", "id": "medusa", "path": "skills/foo/SKILL.md", "hash": "sha256:..." },
        { "kind": "navori-version", "value": "0.2.0" }
      ]
    }
  }
}
```

Cada output declara las fuentes que pueden invalidarlo. `render` calcula hash de cada source actual y compara con el cache. **Solo re-render outputs cuyas sources cambiaron.**

Sinergía con 3.6.2: la sección `sources[]` de cada output coincide con el provenance file (3.6.2). El cache lee del provenance para saber qué dependencias trackear.

**Complejidad estimada**: ~300 LOC (vs ~120 LOC del approach conservador). El costo se justifica por:
- Re-renders de un solo plugin en un repo grande no tocan los otros outputs.
- Provenance ya viene en v0.2 (Q4), las sources son aprovechamiento del mismo grafo.
- Test E2E sólido: change A no debe invalidar B.

**3.3.2 I/O paralelo**
Escrituras de archivos independientes con `Promise.all`. File reads en `detect.ts` en paralelo (no secuenciales). Sospecho que hoy es secuencial en varios puntos.

> **v0.2 — NO SE CONSTRUYE.** El sospechado cuello existe (writes con `fsync`
> secuenciales = 62ms del render) pero el total ya está 7× bajo target.
> Paralelizar exige volver async `writeFileAtomic` y propagar async por todo el
> flujo sync hasta el handler citty (refactor arquitectónico) para ganar ~40ms.
> No se justifica para tool-for-self. Reabrir si un repo pesado lo pide.

**3.3.3 Lazy plugin loading**
No parsear todos los plugin manifests al startup. Cargar solo los referenciados en `navori.config.json`. Save N file reads para CLIs que tengan plugins externos.

**3.3.4 `NAVORI_BENCH=1` con timings**
Emite a stderr una tabla:
```
detect      120ms
loadConfig   15ms
loadPresets  45ms
render      280ms
total       460ms
```
Permite identificar regresiones de performance en CI futuro.

**3.3.5 Single-pass config validation**
Validar `navori.config.json` una vez al inicio, pasar el objeto validado por referencia. No revalidar en cada subcommand.

> **v0.2 — YA CUMPLIDO.** El mapeo confirma que `readConfig` (que hace
> `NavoriConfigSchema.parse`) se llama una sola vez en `runRender` y el objeto
> validado se pasa por referencia a `renderClaudeEngine` / `effectiveConfigForWorkspace`.
> No hay re-validación en subpasos. Sin cambios necesarios.

**Targets cuantitativos:**
- `navori init` < 2s en repo limpio.
- `navori render` < 500ms en repo chico (services--*).
- `navori render` < 2s en `bonum-webapp` (caso pesado).
- `navori doctor` < 1s.

### 3.4 Code quality CLI internals (P1)

**3.4.1 Test coverage target**
80% en `packages/cli/src/lib/`. Hoy no medido. Agregar `vitest --coverage` a CI con threshold.

**3.4.2 Schemas Zod con tests de boundary**
Cada schema tiene test:
- Caso válido pasa.
- Cada campo requerido faltante falla.
- Tipos incorrectos fallan.
- Valores fuera de rango fallan.

Previene drift entre schema y validación implícita.

**3.4.3 Helpers consolidados en `lib/util/`**
Auditar `packages/cli/src/lib/` por funciones duplicadas (path manipulation, fs ops, hashing, diff). Consolidar. Reduce surface de mantenimiento.

**3.4.4 Logger estructurado**
```ts
import { log } from './lib/util/log';
log.debug('rendering', { file, hash });
log.warn('drift detected', { path });
```
Controlado por `NAVORI_LOG=debug|info|warn|error`. Reemplaza `console.log` ad-hoc.

**3.4.5 Errores tipados**
Clases por dominio:
- `ConfigValidationError` — schema fail.
- `RenderError` — falla al escribir.
- `DriftError` — sync detecta drift no resolvible.

Cada una con `code: string` para CI parsing.

**3.4.6 Lint estricto**
`no-any` enforced. `prefer-const`. `no-unused-vars`. ESLint en CI fail si rompe.

**3.4.7 Bundle size tracking**
`dist/index.js` tamaño tracked en CI. Threshold: 200kb. Si crece, alerta.

### 3.6 Patrones / arquitectura (P2)

**3.6.1 Separación plan vs apply (de ECC, light version)**
`render` se split:
- `navori render --plan` → emite JSON con la lista de cambios planeados. No escribe.
- `navori render --apply` → ejecuta el plan.
- `navori render` (default) → plan + apply en uno (UX actual).

Permite que CI corra `--plan --json` y haga assertions sin mutar disk.

**3.6.2 Provenance tracking en archivo aparte, commited**
Decisión Q4: archivo separado `.navori/provenance.json`, commited al repo (NO gitignored).

```json
{
  "version": 1,
  "outputs": {
    "CLAUDE.md": {
      "plugin": "core",
      "version": "0.2.0",
      "sources": ["navori.config.json#core", "presets/medusa/extras/CLAUDE.md.tpl"]
    },
    ".claude/skills/medusa/SKILL.md": {
      "plugin": "preset:medusa",
      "version": "1.0.0",
      "sources": ["presets/medusa/skills/medusa/SKILL.md"]
    }
  }
}
```

Permite:
- `navori uninstall <plugin>` preciso (sabe exactamente qué archivos borrar).
- Auditoría git-historable: "¿desde cuándo este SKILL.md viene de este plugin?".
- Sync sabe qué managed regions pertenecen a qué plugin.
- Input al cache invalidation graph (3.3.1): `sources[]` define las dependencias para hash check.

**Por qué archivo aparte y no en navori.config.json**:
- El config queda lean y user-editable.
- Provenance cambia en cada render; ruido de diff aislado.
- Auditoría git de provenance separada de auditoría de intent (config).

**Por qué commited y no gitignored**:
- Auditoría histórica (git blame).
- Reproducibilidad: nuevo clon llega con provenance al día.
- `sync` puede confiar en él sin re-derivar desde cero.

Posible escalamiento futuro: si el diff de provenance ensucia los PRs, evaluar un `.gitattributes` con `linguist-generated=true` para colapsarlo en GitHub UI.

**3.6.3 Plugin manifest con Zod + JSON Schema generado**
Mantener Zod como source. Build step (`pnpm build:schemas`) corre `zod-to-json-schema` y publica `schemas/plugin.schema.json`. Doble DX (Zod en código, JSON Schema para tooling externo si alguna vez aplica).

**3.6.4 Hooks defensivos por default**
Cuando el render genere hooks, wrappear con `command -v <runtime> >/dev/null 2>&1 && <cmd> || exit 0`. Matchers SessionStart completos por default: `startup|resume|clear|compact`. Cross-platform: emitir siempre `command` + `commandWindows`.

**3.6.5 Recipe "variantes inline filtrables" documentada**
Doc en `docs/recipes/skill-variants.md`: cómo escribir un SKILL.md con modos y cómo los filtra el render. 40 LOC del render + 1 página de doc.

**3.6.6 Single writer principle**
Un solo módulo en la CLI escribe a disk (`lib/render/write.ts`). El resto produce strings/ASTs. Permite:
- Dry-run trivial (skip ese módulo).
- Hash cache en un solo punto.
- Auditoría de side effects.

## 5. Anti-objetivos (filtrados de ECC y ponytail)

Lecciones de auditoría que **no aplican a tu caso** y que se deben rechazar si tentación aparece:

| Anti-objetivo | De dónde viene | Por qué no |
|---|---|---|
| Multi-engine real (5+ engines) | ECC tiene 11 | Solo usas Claude Code. Cuando aparezca otro engine en tu workflow real, entonces. |
| Selective install architecture (profiles/modules/components) | ECC | Sobreingeniería. Tienes 1-3 presets totales. |
| JSON Schemas como producto público | ECC | Zod te alcanza. Genera schema solo si lo consume otra herramienta. |
| Control plane Rust + SQLite + dashboard | ECC `ecc2/` | No orquestas N sesiones. Cuando lo necesites, hablamos. |
| Branding, hero image, sponsors, monetización | ECC + ponytail | No es producto a popularizar. |
| Multi-language docs (12 locales) | ECC | Overhead masivo sin community. Inglés + español MX. |
| Legacy command shims (92 archivos) | ECC | Cuando renombres un comando y rompa tus 12 repos, entonces. Hoy no. |
| Catalog count "270 skills" inflation | ECC bug visible | Counts honestos siempre. |
| Plugin ecosystem público formal | ECC + ponytail | Plugins internos a tu workflow. Si publicas alguno, hablamos. |
| Continuous learning "instincts" YAML | ECC homunculus | Solapamiento con engram que ya usas. |
| Status snapshot como handoff PR-able | ECC `ecc status --markdown` | No tienes handoffs frecuentes a otros devs (todavía). |

## 7. Decisiones tomadas

| Pregunta | Decisión | Implicaciones |
|---|---|---|
| **Q1** ¿`render` UX por default? | **Preview-default, `--apply` para escribir** | Breaking change v0.1→v0.2. Reflejado en 3.1.3. Absorbe 3.6.1 (plan-vs-apply split). |
| **Q2** ¿Invalidación del hash cache? | **Granular por dependencia** | 3.3.1 sube de 120 a 300 LOC. Consume el grafo de sources de 3.6.2 (provenance). Justifica que 3.6.2 sea P1. |
| **Q3** ¿Variantes inline filtrables? | **v0.2 como recipe doc, sin implementar** | 3.2.3 y 3.6.5 ambos cubren el patrón a nivel doc en `docs/recipes/skill-variants.md`. Implementación queda para v0.3 si aparece caso. |
| **Q4** ¿Dónde vive provenance? | **`.navori/provenance.json` aparte, commited** | 3.6.2 documentado con shape concreto. Auditoría git histórica. Reproducible en clones nuevos. |
| **Q5** ¿`doctor --strict` en CI? | **No CI, sí pre-commit hook opt-in scaffolded** | Nuevo item 3.1.7. `navori init` ofrece scaffoldear el hook. Cero fricción CI; opt-in local. |

**Patrón emergente de tus decisiones**: priorizas **seguridad explícita en el output** (preview-always) + **precisión sobre simplicidad** (cache granular, provenance commited) + **cero fricción externa imponiendo CI** (hook opcional). Esto es consistente con tool-for-self: tú controlas el flujo, no necesitas guardrails de equipo, pero quieres garantías estructurales sólidas en lo que el tool produce.
