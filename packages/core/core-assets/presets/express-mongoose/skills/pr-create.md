# pr-create — crear PR contra `{{branchBase}}`

## Cuándo usar este skill

Al cierre del ciclo, tras el `APPROVED` del reviewer. Es la Fase 8 del pipeline `ticket-intake`. El pre-flight es bloqueante: ningún PR sale si el gate no está verde o si falta aprobación cuando el harness está activo.

## Pre-flight (bloqueante — cualquier check falla → ABORT con mensaje claro)

1. **Working tree limpio** o solo cambios del feature: `git status --porcelain`. Si hay cambios uncommitted que NO son del feature, párate y pregunta.
2. **No estás en rama base**: aborta si la rama es `main|master|{{branchBase}}`.
3. **`gh` autenticado**: `gh auth status`.
4. **Gate verde en este turno**: `{{qualityGate.fast}}`. Falla → manda al implementer.
5. **Review APPROVED** (si el harness está activo): `grep -q APPROVED .claude/progress/review_*.md`. Falla → manda al reviewer.
6. **Hay commits pusheables**: `git log origin/{{branchBase}}..HEAD --oneline`. Vacío → nada que PR-ear.
7. **Branch sincronizada**: `git fetch origin {{branchBase}}` y `git log HEAD..origin/{{branchBase}} --oneline`. Si origin va adelante, considera rebase y pregunta al usuario.

## Push y redacción

Si la branch no está pusheada: `git push -u origin "$current_branch"`. Si el upstream diverge, **no fuerces push** — pregunta.

Delega title + body a un `Agent` (modelo rápido). Reglas que le impones:
- **Title**: Conventional Commit con scope de tu dominio, ≤70 chars, español MX, sin emojis ni puntuación final.
- **Body**: markdown con secciones fijas `## Resumen`, `## Cambios`, `## Test plan`, `## Notas` (opcional). Tono directo, sin "Generated by Claude".
- **Contexto al redactor**: branch, base `{{branchBase}}`, ticket, `impl_<feature>.md`, `review_<feature>.md`, `git log {{branchBase}}..HEAD --oneline` y `--stat`.
- **Output**: JSON exacto `{"title": "...", "body": "..."}` sin fences.

Valida el output: `title` ≤70 y matchea `^(feat|fix|refactor|chore|docs|style|test|build|ci|perf)(\(.+\))?: `; `body` contiene Resumen, Cambios y Test plan. Si falla, re-invoca al redactor con el feedback (máx. 2 reintentos).

## Crear el PR

```bash
gh pr create --base {{branchBase}} --title "$title" --body "$body"
```

Captura la URL del output y muéstrasela al usuario.

## Reglas duras

- **NUNCA `git push --force`** sin que lo pida el usuario.
- **NUNCA `--base main` ni `--base master`.** Siempre `--base {{branchBase}}`.
- **NUNCA `--no-verify`** en commits previos.
- **Cambios uncommitted ajenos al feature** → párate y pregunta, no los arrastres al PR.
- **No agregues** "Generated with Claude" ni emojis al body.
- **Reviewers/labels**: no los agregues automático salvo que el usuario lo pida.

## Antes de declarar listo

- El pre-flight pasó completo (gate verde, branch ≠ base, commits pusheables, review APPROVED si aplica).
- El title pasó la validación Conventional Commit (≤70 chars) y el body tiene las secciones fijas.
- El PR se creó contra `{{branchBase}}` y su URL quedó mostrada al usuario.
- Post-PR: `mem_save` de cualquier decisión no obvia, entrada en `history.md`, `current.md` a `state: idle`, y `mem_session_summary`.
